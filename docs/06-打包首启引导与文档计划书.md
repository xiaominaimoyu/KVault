# 阶段 6：打包、首启引导与文档计划书

> 对应目标：把项目从“开发环境可运行”整理为“可长期使用、可迁移、可交付”的桌面应用。
> 建议周期：第 5 周
> 阶段性质：发布准备阶段

---

## 1. 阶段目标

本阶段完成应用打包、首次启动检查、配置引导、日志完善和使用文档。完成后，用户应能在不理解源码结构的情况下运行应用，并知道如何配置模型、导入文档、使用检索和连接 MCP。

## 3. 打包规划

### 3.1 PyInstaller 配置

新增：

```text
build/
└── pyinstaller.spec
```

或在根目录维护：

```text
KVault.spec
```

需要处理：

- PySide6 Qt 插件。
- assets 资源。
- config 模板。
- MCP server 模块。
- ChromaDB 运行依赖。

验收标准：

- 可生成 Windows `.exe`。
- 打包后应用可启动主窗口。
- 打包后可导入文档。
- 打包后可检索已有知识库。

### 3.2 路径策略

开发环境路径和打包环境路径必须分开处理。

建议：

- 应用程序目录只放程序文件。
- 用户数据放在可配置目录。
- 默认数据目录可放在项目目录或用户目录，第一版可继续使用 `./data`，但文档中说明。

验收标准：

- 打包后不会尝试写入只读程序目录。
- 用户可以在设置中看到数据目录。

## 4. 首次启动引导

首次启动时检查：

- `config.json` 是否存在。
- `data/files/` 是否存在。
- `data/chroma_db/` 是否存在。
- `data/kb.sqlite` 是否存在。
- Ollama 服务是否可访问。
- Embedding 模型是否可调用。

建议首启流程：

```text
检查配置
→ 检查数据目录
→ 检查 SQLite
→ 检查 Ollama
→ 检查模型
→ 显示可继续使用或需要处理的问题
```

验收标准：

- 缺少配置时自动从模板创建。
- 缺少数据目录时自动创建。
- Ollama 未启动时提示用户启动。
- 模型不可用时提示模型名称和解决方向。

## 5. 错误处理与日志完善

日志文件：

```text
data/logs/app.log
```

日志级别：

- INFO：启动、导入成功、检索请求。
- WARNING：配置缺失、模型不可用、非关键失败。
- ERROR：导入失败、数据库失败、MCP 启动失败。

GUI 提示原则：

- 用户看到简短可理解的信息。
- 技术细节写入日志。
- 关键失败提供下一步行动建议。

验收标准：

- 常见错误不会只显示 traceback。
- 日志中能定位失败模块和异常信息。

## 7. 依赖锁定

将开发阶段的宽松依赖整理为可复现版本。

最低输出：

- `requirements.txt`
- 可选 `requirements-dev.txt`

建议区分：

```text
requirements.txt       # 运行依赖
requirements-dev.txt   # 测试、打包、开发依赖
```

验收标准：

- 新环境按文档安装依赖后可运行。
- 依赖版本不会因为自动升级导致明显不兼容。

- 执行检索。
- 检查日志。

- 通过命令启动 stdio MCP。
- 外部 Client 调用三个工具。
- 确认返回结构符合文档。

## 10. 风险与控制

| 风险 | 处理方式 |
|------|----------|
| PySide6 打包体积大 | 第一版接受体积，后续再考虑 UPX 或 Nuitka |
| 打包后路径错误 | 统一封装路径解析，不在业务代码中散落相对路径 |
| 用户不知道模型未启动 | 首启检查和状态栏明确显示 Ollama 与模型状态 |
| 数据迁移困难 | 明确说明必须备份 `data/files/`、`data/chroma_db/`、`data/kb.sqlite` |
