# 阶段 2：真实索引与元数据管理计划书

> 对应目标：把阶段 1 的“能入库”升级为“可追踪、可恢复、可管理”的真实索引系统。
> 建议周期：第 2 周
> 阶段性质：核心数据层完善

---

## 1. 阶段目标

本阶段要接入真实 Embedding 服务，并引入 SQLite 元数据管理。完成后，系统不仅能把文本块写入 ChromaDB，还能知道每个文档的来源、状态、分区、标签、导入时间和索引结果。

本阶段完成后，重启应用时应能恢复文档列表，而不是只依赖当前内存状态。

---

## 2. 本阶段范围

### 2.1 必须完成

- 实现 `EmbeddingService`。
- 接入 Ollama Embedding。
- 实现 SQLite 元数据管理模块。
- 建立文档、chunk、分区、标签基础表。
- 完善导入状态流转。
- 实现后台导入线程，避免 GUI 卡死。
- 文档列表展示真实元数据。
- 右侧预览区显示解析文本与 chunk 概览。

### 2.2 暂不完成

- 不做复杂检索排序调优。
- 不做 MCP。
- 不做打包。
- 不做完整主题系统。
- 不做云端 Embedding 多供应商切换。

---

## 3. 数据模型规划

### 3.1 documents 表

建议字段：

| 字段 | 说明 |
|------|------|
| `id` | 文档唯一 ID |
| `file_name` | 文件名 |
| `original_path` | 原始路径 |
| `stored_path` | 复制后的本地存储路径 |
| `file_ext` | 文件扩展名 |
| `file_size` | 文件大小 |
| `partition_id` | 所属分区 |
| `status` | `pending/indexing/indexed/failed` |
| `chunk_count` | 文本块数量 |
| `error_message` | 失败原因 |
| `created_at` | 导入时间 |
| `updated_at` | 更新时间 |

### 3.2 chunks 表

建议字段：

| 字段 | 说明 |
|------|------|
| `id` | chunk 唯一 ID |
| `document_id` | 所属文档 |
| `chunk_index` | 块序号 |
| `content_preview` | 片段预览 |
| `token_count` | 估算 token 数 |
| `chroma_id` | ChromaDB 中的 ID |

### 3.3 partitions 表

第一版内置默认分区：

- 全部文档
- 技术笔记
- 项目资料
- 阅读笔记
- 灵感收藏

### 3.4 tags 与 document_tags 表

本阶段只需要提供数据结构和基础 CRUD，复杂标签 UI 可放到阶段 4 完善。

## 4. 任务拆解

### 4.1 Embedding 服务

新增 `core/embedding_service.py`。

最低接口：

```python
class EmbeddingService:
    def embed_texts(self, texts: list[str]) -> list[list[float]]:
        ...

    def embed_query(self, query: str) -> list[float]:
        ...

    @property
    def dimension(self) -> int:
        ...
```

默认配置：

- provider：`ollama`
- model：`bge-large-zh-v1.5`
- base_url：`http://localhost:11434`
- batch_size：`32`

验收标准：

- 服务启动时可检测 Ollama 是否可访问。
- 批量文本能返回同维度向量。
- 模型不可用时返回明确错误，不让 GUI 直接崩溃。

### 4.2 SQLite 元数据管理

新增 `core/metadata_manager.py`。

最低能力：

- 初始化数据库与表结构。
- 新增文档记录。
- 更新索引状态。
- 写入 chunk 元数据。
- 查询文档列表。
- 查询单个文档详情。
- 删除文档元数据。

验收标准：

- 应用首次启动时自动创建 `data/kb.sqlite`。
- 导入文档后 documents 表有记录。
- 导入失败时 status 为 `failed`，并保存错误原因。

### 4.3 完整导入状态流转

导入流程调整为：

```text
创建 documents 记录 pending
→ 复制原始文件
→ status = indexing
→ 解析文本
→ 切分 chunk
→ 生成真实 embedding
→ 写入 ChromaDB
→ 写入 chunks 表
→ status = indexed
```

失败流程：

```text
任何步骤异常 → status = failed → 记录 error_message
```

验收标准：

- 任一步失败都不会产生“看起来成功但实际不可用”的文档。
- 重复导入同名文件时有明确策略：允许生成新 ID，暂不做去重。

### 4.4 后台导入线程

新增 `gui/workers/ingest_worker.py`。

信号建议：

- `progress(percent, message)`
- `file_done(document_id, success)`
- `finished(success_count, fail_count)`

验收标准：

- 批量导入时 GUI 不假死。
- 状态栏能显示当前处理文件。
- 导入完成后文档列表刷新。

### 4.5 文档列表绑定真实数据

完善中间文档列表。

显示列：

- 文件名
- 格式
- 大小
- 导入时间
- 状态
- chunk 数

验收标准：

- 应用重启后列表能从 SQLite 恢复。
- 状态颜色能区分已索引、索引中、失败、待处理。

### 4.6 预览面板基础版

右侧预览区先显示：

- 文档名称
- 文件格式
- chunk 数
- 解析后的纯文本前若干字符
- chunk 列表或当前 chunk 预览

验收标准：

- 点击文档列表中的文档，右侧预览刷新。
- 无预览内容时显示空状态，而不是报错。

## 6. 风险与控制

| 风险 | 处理方式 |
|------|----------|
| Embedding 模型过大或不可用 | 允许配置小模型作为开发替代 |
| 导入过程中部分写入成功、部分失败 | 使用状态字段和失败记录保证可恢复 |
| ChromaDB 与 SQLite 数据不一致 | 每个 chunk 保存 `document_id` 和 `chroma_id`，删除和重建时按文档 ID 操作 |
| 后台线程更新 UI 出错 | 只通过 Qt Signal 回到主线程更新界面 |

---

## 7. 完成定义

当系统可以批量导入文档，使用真实 Embedding 生成向量，并在重启后恢复完整文档列表和索引状态时，本阶段完成。
