# 个人知识库系统 — 开发方案

> **版本**: v1.0  
> **日期**: 2026-07-13  
> **架构**: PySide6 桌面 GUI + LangChain RAG 后端 + MCP 对外接口  
> **定位**: 个人自用本地知识库，支持多格式文档导入、向量化存储、可视化管理和外部 Agent 接入

---

## 一、项目概述

### 1.1 目标与愿景

构建一套完全本地运行的个人知识管理系统，核心能力包括：

- **多格式文档导入**：支持 PDF、Word（DOCX）、Markdown、TXT、Excel、PPT 等常见格式
- **智能解析与向量化**：自动完成文档解析 → 文本切分 → Embedding 向量化 → 入库全流程
- **可视化桌面管理**：基于 PySide6 的三栏式 GUI，提供分区导航、文档列表、预览与检索测试
- **外部 Agent 接入**：通过 MCP（Model Context Protocol）标准协议，供外部 AI Agent 调用检索

### 1.2 核心原则

| 原则 | 说明 |
|------|------|
| 本地优先 | 全部组件可本地运行，数据不出内网 |
| 渐进增强 | 从最小可用版本起步，按需迭代 |
| 解耦设计 | GUI / 后端服务 / MCP 接口三层分离 |
| 标准协议 | 采用 MCP 协议对外暴露检索能力 |

## 二、技术选型总览

### 2.1 技术栈一览

| 层级 | 选型 | 版本/规格 | 说明 |
|------|------|----------|------|
| **桌面 GUI** | PySide6 (Qt6) | ≥ 6.4 | LGPL 许可证，Qt 官方维护 |
| **RAG 编排框架** | LangChain | ≥ 0.3 | 文档加载、切分、向量管道编排 |
| **PDF 解析** | PyMuPDF (fitz) | latest | 结构化 PDF 文本提取 |
| **Word 解析** | python-docx | latest | 保留标题层级信息 |
| **复杂 PDF 解析**（可选） | Docling (IBM) | latest | 表格/公式/版式高精度解析 |
| **文本切分器** | RecursiveCharacterTextSplitter | — | 块大小 500 字符，重叠 100 字符 |
| **Embedding 模型** | bge-large-zh-v1.5 | — | 中文语义理解，本地部署 |
| **Embedding 运行时** | Ollama | latest | 本地模型推理服务 |
| **向量数据库** | ChromaDB | ≥ 0.4 | 轻量级，自带持久化与元数据过滤 |
| **元数据存储** | SQLite | 内置 | 文档信息、分区、标签管理 |
| **对外接口协议** | MCP (Model Context Protocol) | — | JSON-RPC 2.0 双向通信 |
| **HTTP 服务**（可选演进） | FastAPI | ≥ 0.100 | GUI 与后端解耦时的中间层 |
| **打包分发** | PyInstaller / Nuitka | latest | 打包为单可执行文件 |

### 2.2 选型决策记录

#### 为什么选择 PySide6 而非其他 GUI 框架？

| 框架 | 优势 | 劣势 | 结论 |
|------|------|------|------|
| **PySide6** | LGPL 免费商用；Qt 官方维护；组件体系完整；跨平台一致性好；原生支持 PDF 预览（6.4+） | 学习曲线中等 | ✅ **采用** |
| PyQt6 | 社区资源丰富；功能成熟 | GPL 许可证闭源需付费购买商业许可 | ❌ 许可风险 |
| Tkinter | Python 标准库零安装 | 组件基础无法满足富文本预览等需求；界面陈旧 | ❌ 能力不足 |
| DearPyGui | 高性能 Immediate Mode 渲染 | 更适合数据可视化和实时图形场景；文档管理非最佳匹配 | ❌ 场景不匹配 |
| Flet | 现代 Flutter 风格 UI | 底层依赖 Web 技术，非原生桌面体验 | ❌ 不符合"不用 Web"要求 |

#### 为什么选择 ChromaDB 作为向量数据库？

| 向量数据库 | 特点 | 适用场景 | 结论 |
|-----------|------|---------|------|
| **ChromaDB** | 纯 Python；API 极简；自带持久化；LangChain/LlamaIndex 原生集成 | 个人知识库、快速原型 | ✅ **采用** |
| LanceDB | 列式存储；零配置；支持多模态 | 需要图片/音频向量的场景 | 备选 |
| Qdrant | Rust 引擎；高性能；支持混合搜索 | 数据量增长后性能升级路径 | 演进备选 |
| Milvus | 分布式；海量数据支持 | 企业级生产集群 | 过重 |

### 3.2 数据流

**写入流（文档导入）**

```
用户拖拽/选择文件 → GUI 触发导入事件
    → QThread 异步执行:
        1. 文件复制到 data/files/
        2. 根据格式分发到对应解析器 (PyMuPDF/python-docx/原生)
        3. 提取纯文本内容
        4. RecursiveCharacterTextSplitter 切分为 chunk
        5. 调用 Ollama (bge-large-zh-v1.5) 生成向量
        6. 写入 ChromaDB (向量 + 文本块 + 元数据)
        7. 写入 SQLite (文件名/格式/大小/时间/分区/标签/索引状态)
    → 进度信号回传 GUI 更新进度条和状态标签
```

**读取流（检索查询）**

```
GUI 输入查询 / MCP Server 收到请求
    → 问题文本向量化 (同一 Embedding 模型)
    → ChromaDB 相似度搜索 (余弦相似度, Top-K)
    → 返回排序后的文本块列表 (含相似度分数 + 来源文档元数据)
    → GUI 展示结果 / MCP 协议返回给 Agent
```

### 5.2 核心模块 API 设计

#### 5.2.1 文档解析器 (`document_parser.py`)

```python
class DocumentParser:
    """多格式文档解析器，统一输出纯文本 + 元信息"""

    def parse(self, file_path: str) -> ParsedDocument:
        """
        根据文件扩展名自动选择解析器
        
        返回:
            ParsedDocument(
                content: str,           # 提取的纯文本
                metadata: dict,         # {title, author, page_count, ...}
                sections: list[Section], # 结构化章节（如有）
            )
        
        支持格式:
            .pdf  → PyMuPDF (fitz)，扫描件走 OCR 兜底
            .docx → python-docx，保留标题层级
            .md   → 原生读取
            .txt  → UTF-8 读取
            .xlsx → openpyxl 逐表提取
            .pptx → python-pptx 逐页提取
        """

    def supported_formats(self) -> list[str]:
        """返回支持的文件扩展名列表"""
```

#### 5.2.2 文本切分器 (`text_splitter.py`)

```python
class KnowledgeTextSplitter:
    """基于 LangChain RecursiveCharacterTextSplitter 的封装"""

    def __init__(self, chunk_size: int = 500, chunk_overlap: int = 100):
        self.splitter = RecursiveCharacterTextSplitter(
            chunk_size=chunk_size,
            chunk_overlap=chunk_overlap,
            separators=["\n\n", "\n", "。", "！", "？", " ", ""]
        )

    def split(self, content: str, doc_metadata: dict) -> list[Chunk]:
        """
        将长文本切分为语义块
        
        返回:
            Chunk(id, content, metadata, token_count)
        
        每个 chunk 自动携带:
            - source_file: 来源文件名
            - chunk_index: 块序号
            - total_chunks: 总块数
            - partition: 所属分区
            - tags: 继承自文档的标签
        """
```

### 5.3 异步任务设计

所有耗时操作必须通过 `QThread` 在后台线程执行：

```python
# gui/workers/
class DocumentIngestWorker(QThread):
    """文档导入与索引工作线程"""
    progress = Signal(int, str)       # (百分比, 当前文件名)
    file_done = Signal(str, bool)     # (文件路径, 是否成功)
    finished = Signal(int, int)       # (成功数, 失败数)

    def __init__(self, file_paths: list[str]):
        super().__init__()
        self.file_paths = file_paths

    def run(self):
        total = len(self.file_paths)
        success_count = 0
        fail_count = 0
        for i, path in enumerate(self.file_paths):
            try:
                self.progress.emit(int((i+1)/total*100), os.path.basename(path))
                # 调用 core 层完成: 解析 → 切分 → 向量化 → 入库
                ingest_single_document(path)
                self.file_done.emit(path, True)
                success_count += 1
            except Exception as e:
                self.file_done.emit(path, False)
                fail_count += 1
        self.finished.emit(success_count, fail_count)


class SearchWorker(QThread):
    """异步检索工作线程"""
    results_ready = Signal(list)  # SearchResult 列表

    def __init__(self, query: str, top_k: int = 5, filters: dict = None):
        super().__init__()
        self.query = query
        self.top_k = top_k
        self.filters = filters

    def run(self):
        results = retriever.search(self.query, self.top_k, self.filters)
        self.results_ready.emit(results)
```

## 六、MCP Server 对外接口设计

### 6.1 协议概述

MCP（Model Context Protocol）是基于 JSON-RPC 2.0 的双向通信协议，让 AI Agent 能够标准化地连接和使用外部工具。

**角色分工**：
- **MCP Server（本项目）**：将知识库检索能力封装为标准工具
- **MCP Client（外部 Agent）**：Claude Desktop / Cursor / 自建 Agent 等
- **通信方式**：stdio（标准输入输出）或 SSE（HTTP 流）

## 十、风险与应对

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|---------|
| 大文件（>50MB PDF）解析耗时过长 | 用户等待体验差 | 中 | 拆分为分页解析 + 流式进度反馈；设置文件大小上限 |
| Embedding 模型显存不足 | 无法本地运行向量化 | 低 | 提供云端 API 替代选项（OpenAI / 通义千问）；使用更小的模型如 bge-small |
| ChromaDB 数据量增长后性能下降 | 检索延迟增加 | 低 | 设定性能监控阈值；预留迁移至 Qdrant 的路径 |
| MCP 协议版本迭代不兼容 | 外部 Agent 无法连接 | 低 | 锁定 MCP 版本号；关注上游更新及时跟进 |
| PySide6 打包体积过大 | 分发不便 | 中 | 使用 UPX 压缩；仅打包必要 Qt 模块；考虑 Nuitka 替代 |
